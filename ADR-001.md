# ADR-001: Initial Architecture

- **Status**: Accepted
- **Date**: 2026-05-15

## Context

We are building a Rustlings/koans-style web application for learning Haskell, where users complete exercises checked by running their submitted code against a hidden test suite. The application has three components: a Haskell API backend, a browser-based frontend with an embedded code editor, and a sandboxed code execution service. The project is maintained by a single developer (senior Ruby/Rails engineer, actively learning Haskell) with approximately 5–10 hours per week available. The target is a demonstrable MVP in roughly 3 months. These decisions are load-bearing because they determine framework lock-in, operational overhead, and the learning investment required before any feature work can begin.

## Decisions

### Backend Framework: Servant

**Decision:** Servant, with Wai/Warp as the underlying server.

**Alternatives considered:** Scotty (simpler, Sinatra-like), Yesod (Rails-like conventions, heavy Template Haskell), IHP (opinionated full-stack).

**Reasoning:** Servant's type-level routing encodes the API contract in the type system, catching mismatches at compile time and teaching idiomatic Haskell design. It is the most common Haskell web framework in production, giving the project the best long-term ecosystem support. The initial learning curve is steeper than Scotty, but the friction is productive for the stated goal of learning Haskell.

### Frontend: TypeScript SPA (Vite + React or Svelte)

**Decision:** A thin TypeScript + Svelte SPA calling the Haskell API, hosted on Vercel or Netlify.

**Alternatives considered:** Haskell-to-JS (GHCJS, Miso, Reflex), server-rendered MPA, React.

**Reasoning:** Keeps the frontend in familiar territory, concentrating Haskell learning on the backend. Svelte chosen over React for its simpler component model — less boilerplate reduces cognitive load when the primary learning investment is already on the Haskell side. Haskell-to-JS frameworks add a significant secondary learning burden with marginal product benefit.

### Code Editor: CodeMirror 6

**Decision:** CodeMirror 6 as the in-browser code editor component.

**Alternatives considered:** Monaco (~4–5MB, VS Code engine, poor mobile), plain textarea (no syntax highlighting).

**Reasoning:** CodeMirror 6 is ~250KB, modular, mobile-friendly, and has adequate Haskell syntax support. Monaco's bundle size and poor mobile story are unnecessary for an exercise app where users write small snippets.

### Database and Access Layer: PostgreSQL + Persistent + Esqueleto

**Decision:** PostgreSQL as the database; Persistent for schema definition and migrations; Esqueleto for type-safe queries.

**Alternatives considered:** SQLite (simpler ops, less scalable), raw `postgresql-simple` (less type safety).

**Reasoning:** The most documented and commonly used Haskell web stack combination. Managed Postgres via Supabase or Neon free tier during development; upgrade to paid tier at launch.

### Code Execution Sandbox: Managed Judge0 Cloud

**Decision:** Judge0 cloud (managed, via RapidAPI pay-per-use at $0.0017/submission; intro volume tier $44.99/month, break-even ~882 submissions/day). Deferred until the sandbox integration phase; development proceeds without it.

**Alternatives considered:** Self-hosted Judge0 (lower cost, high ops burden), Piston (simpler API, less battle-tested for Haskell), custom Docker sandbox (maximum control, maximum work).

**Reasoning:** Solo maintainer time is better spent on the product than on sandbox infrastructure. The price delta vs. self-hosting (~$10–25/month) does not justify the ongoing operational overhead. Judge0 is deferred until needed, keeping development costs near zero until launch.

### Authentication: GitHub OAuth via Clerk

**Decision:** GitHub OAuth only at MVP, via Clerk (free tier). Authenticated access required to submit code. Email/password and GitLab OAuth deferred.

**Alternatives considered:** Auth0, rolling own JWT auth, anonymous access.

**Reasoning:** The target audience (developers learning Haskell) almost universally has a GitHub account. Managed auth via Clerk eliminates the complexity of password flows, reset emails, and secure credential storage. Requiring auth gates code execution, reducing sandbox abuse surface.

### Deployment: Fly.io (PaaS)

**Decision:** Fly.io for the API server; Vercel or Netlify for the frontend. Postgres via Neon or Supabase free tier.

**Alternatives considered:** Railway (simpler DX but higher cost at ~$20–40/month with managed Postgres), VPS (lower cost, higher ops), AWS/GCP (overkill at this scale).

**Reasoning:** A solo maintainer without time for infrastructure management needs git-push deploys, automatic TLS, and managed restarts. Fly.io chosen for cost predictability: a small shared machine (~$2–5/month) + external managed Postgres free tier keeps running costs near zero during development and low at launch. Docker-native workflow suits the existing Dockerfile well.

### Exercise Content Authoring: Git-Based Flat Files

**Decision:** Exercises stored as structured files in the repository (description, stub code, hidden test suite, canonical solution, hints array). New exercises added via pull request.

**Alternatives considered:** Admin UI with database-backed authoring, seed files only.

**Reasoning:** Enables community contributions via standard PR workflow. Keeps admin UI out of MVP scope. Version-controlled content is straightforward to audit and roll back.

## Consequences

- **Servant learning curve:** Type errors in Servant can be verbose and opaque early on. Expect 1–2 sessions lost to compiler friction before patterns become familiar.
- **GHC compile latency:** Cold compilation of user submissions will take 5–30 seconds. Warm caching strategies are deferred; this will be a noticeable UX limitation at launch.
- **Judge0 third-party dependency:** Execution cost scales with usage. A traffic spike without rate limiting could produce unexpected charges.
- **GitHub-only auth:** Excludes the small population of developers without GitHub accounts. Acceptable for MVP; revisit if it becomes a meaningful barrier.
- **Git-based exercises require a redeploy** to publish new content. Acceptable at current scale; may become friction if exercise authorship grows.

## Review Triggers

- **Switch backend framework** if Servant type errors consistently block feature progress for more than one session per week after the first month.
- **Re-evaluate sandbox** if Judge0 p95 submission latency exceeds 15 seconds, OOM errors appear under real load, or monthly sandbox cost exceeds $80.
- **Add email/password auth** if non-GitHub users represent a meaningful, documented demand.
- **Add admin UI for exercises** if the exercise library exceeds ~100 entries and git-based authoring becomes a bottleneck.
- **Move to VPS or self-hosted Postgres** if total monthly infrastructure cost consistently exceeds $150.
